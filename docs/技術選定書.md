## 2. 各技術の選定理由

### 2.1 Next.js (App Router)

**選定理由:**
- うおじゅんのフルスタック技術選好と合致
- App Routerでサーバーコンポーネント/Route Handlersが使える → APIとフロントが1プロジェクトで完結
- Vercelとの相性が最高（デプロイ0設定）
- エコシステムが成熟、情報量が多い

**代替候補と不採用理由:**
- Remix → 機能的には十分だが、Vercelデプロイの最適化がNext.jsの方が上
- SvelteKit → 学習コスト。エコシステムの成熟度

### 2.2 Supabase

**選定理由:**
- PostgreSQL + Auth + Edge Functions + RLS がオールインワン → 個別構築より圧倒的に速い
- ポイント管理に必要なトランザクション整合性をPostgreSQL関数で担保できる
- RLSで行レベルのアクセス制御が宣言的に書ける
- 無料枠がMVP規模に十分（500MB, Auth無制限）
- TypeScript型自動生成（`supabase gen types`）

**代替候補と不採用理由:**
- Firebase → NoSQL。ポイント管理のトランザクション整合性がFirestoreでは担保しにくい
- PlanetScale → Auth/Edge Functionsが別途必要。統合の手間
- 自前PostgreSQL (AWS RDS) → 管理コストが高い。MVPには過剰

### 2.3 Vercel

**選定理由:**
- Next.jsの開発元。デプロイ最適化が最も進んでいる
- プレビューデプロイ（PRごとに自動デプロイ）が便利
- 無料枠がMVPに十分
- Supabase連携が公式にサポートされている

### 2.4 shadcn/ui + Tailwind CSS

**選定理由:**
- コピペベースのUIコンポーネント → vendor lock-inなし、カスタマイズ自由
- Tailwind CSSとの統合が自然
- ブランドカラー（オレンジ）への変更が`tailwind.config`で一括設定可能
- Radix UIベースでアクセシビリティ対応済み

### 2.5 Resend + React Email

**選定理由:**
- 無料枠: 月3,000通（MVP規模に十分）
- React Emailでテンプレートをコンポーネントとして書ける → TypeScriptの恩恵
- Next.js / Edge Functionsとの相性が良い
- APIがシンプル（1エンドポイント）

**代替候補と不採用理由:**
- SendGrid → 無料枠は100通/日。機能は豊富だがMVPには過剰
- Amazon SES → 設定が複雑。MVPには向かない

## 3. 外部連携の設定方針

### 3.1 Google Calendar API

**認証方式:** サービスアカウント

- かずみんのGoogleカレンダーに対してサービスアカウントに読み取り+書き込み権限を付与
- ユーザーのOAuthトークン管理が不要 → 実装がシンプル
- 祝日判定はGoogle公開カレンダー（`ja.japanese#holiday@group.v.calendar.google.com`）を参照

**使用するAPI:**
- `calendar.freebusy.query` → 空き/ビジー情報取得
- `calendar.events.insert` → 予約時にカレンダーイベント追加
- `calendar.events.delete` → キャンセル時にカレンダーイベント削除
- `calendar.events.list`（祝日カレンダー） → 祝日判定

### 3.2 Zoom API

**認証方式:** Server-to-Server OAuth App × 2アカウント

- Zoom Marketplace でアプリ作成 → `account_id`, `client_id`, `client_secret` を取得
- アカウントA（有料）とアカウントB（無料）で別々のクレデンシャルを環境変数に設定
- メニュー（`meeting_menus`）の `zoom_account_key` で使用するアカウントを切り替え

**使用するAPI:**
- `POST /users/me/meetings` → ミーティング作成
- `DELETE /meetings/{meetingId}` → ミーティング削除

### 3.3 Googleカレンダーに追加ボタン（顧客向け）

ユーザーのカレンダーへの追加はAPIではなく、URLスキームで実現する。

```
https://calendar.google.com/calendar/render?action=TEMPLATE
  &text={セッション名}
  &dates={開始時刻}/{終了時刻}
  &details={Zoomリンク等}
  &location=Zoom
```

- OAuth権限が不要 → Google審査なし
- 全ユーザー（Google認証・メール認証・ゲスト）共通の実装で済む

## 6. 技術構成レビュー結果

### 6.1 この構成は適切か？

**結論: 適切。MVPに過剰ではない。**

| 観点 | 評価 |
|---|---|
| 開発者スキルとの相性 | フルスタック × Next.js選好と完全に合致 |
| 開発速度 | Supabaseのオールインワン構成で個別構築より圧倒的に速い |
| コスト | 両方とも無料枠あり。ユーザー10人・週3〜5件なら$0 |
| ポイント管理の整合性 | PostgreSQLのトランザクション + 関数で担保可能 |

### 6.2 もっと軽量な選択肢ではダメか？

| 代替案 | 不採用理由 |
|---|---|
| Google Sheets + GAS | ポイント管理のトランザクション整合性が担保できない |
| Notion + Zapier | 同上。RLS的なアクセス制御も不可能 |
| ノーコード (Bubble等) | Zoomアカウント使い分け等の独自要件に対応困難 |

**ポイント管理のトランザクション整合性が必要 → RDB (PostgreSQL) は必須。Supabaseはそれを最も手軽に使える手段。**

### 6.3 スケーリング時の注意点

| 懸念 | 発生タイミング | 対策 |
|---|---|---|
| Vercelサーバーレス10秒制限（無料） | 外部API複数呼び出し時 | Pro移行($20/月)で60秒に拡張 |
| Supabase 500MBストレージ（無料） | 予約・ポイント履歴が溜まると到達 | Pro移行($25/月)で8GBに |
| Googleカレンダー同期のオンデマンド方式 | ユーザー100人超でRate Limit | cronバッチに切り替え |
