## 2. 外部連携の実装方針

### 2.1 Googleカレンダー同期（オンデマンド）

```
[顧客が予約ページを開く]
  ↓
[GET /api/slots?date=2026-03-01]
  ↓
[前回同期からの経過時間チェック]
  ├── 15分未満 → DBの既存データを返す
  └── 15分以上 → Google Calendar API呼び出し
        ↓
      [1] calendar.freebusy.query（かずみんのカレンダー、対象日）
      [2] calendar.events.list（祝日カレンダー、対象日）
        ↓
      [3] 対象日が祝日か判定
        ├── 祝日 → weekly_schedulesの'holiday'行を適用
        └── 平日 → weekly_schedulesの対応曜日行を適用
        ↓
      [4] 営業時間内で、busy時間を除外
        ↓
      [5] メニューの最小時間（30分）単位でスロットを生成
        ↓
      [6] 既存の予約（bookings）と重複するスロットを除外
        ↓
      [7] 空きスロット一覧を返却
```

**同期タイムスタンプの管理:**
- Supabaseの `app_settings` テーブル（key-value）に `last_calendar_sync` として保存
- または、シンプルにメモリキャッシュ（Vercelのサーバーレスでは使えないのでDB or KV）

## 3. 予約作成の処理フロー（最重要）

予約作成（`POST /api/bookings`）はシステムの中核。以下の処理を1つのRoute Handlerで統合する。

```
[予約リクエスト受信]
  ↓
[1] バリデーション
    - ゲスト: 名前+メール必須、メニューがゲスト可であること
    - 会員: 認証済み、メニューが会員用、ポイント残高チェック
  ↓
[2] スロット空き確認（バッファ込み）
    - 指定の時間帯 + 前後バッファ範囲に既存の confirmed 予約がないか
    - Googleカレンダーのbusy時間 + バッファと重複していないか
  ↓
[3] ポイント消費（会員のみ）
    - consume_points() PostgreSQL関数を呼び出し
    - 失敗（残高不足）→ 400エラー返却
  ↓
[4] Zoom会議作成
    - メニューのzoom_account_keyに応じたアカウントで生成
    - zoom_meeting_url, zoom_meeting_id を取得
  ↓
[5] 管理者カレンダーにイベント追加
    - Google Calendar API: events.insert
    - google_event_id を取得
  ↓
[6] bookingsテーブルにINSERT
    - 全ての情報をまとめて保存
  ↓
[7] 予約確認メール送信
    - ユーザー宛（Zoomリンク付き）
    - 管理者宛
  ↓
[8] レスポンス返却（booking情報 + Zoomリンク + カレンダー追加URL）
```

**エラー時のロールバック:**
- [3]でポイント消費後に[4]以降で失敗した場合 → refund_points()でポイント返還
- [4]でZoom作成後に[5]以降で失敗した場合 → Zoom会議を削除
- → try/catch + 補償トランザクションで対応

## 4. キャンセルの処理フロー

```
[キャンセルリクエスト受信]
  ↓
[1] 権限チェック（本人 or admin）
  ↓
[2] ポイント返還（会員の場合）
    - refund_points() PostgreSQL関数を呼び出し
  ↓
[3] Zoom会議削除
    - bookings.zoom_account_key でアカウント特定
    - bookings.zoom_meeting_id で会議削除
  ↓
[4] 管理者カレンダーからイベント削除
    - bookings.google_event_id で削除
  ↓
[5] bookings.status = 'cancelled' に更新
  ↓
[6] キャンセルメール送信
    - ユーザー宛 + 管理者宛
  ↓
[7] レスポンス返却
```

## 5. 空き時間算出の詳細ロジック

### 入力

- `date`: 対象日（例: 2026-03-01）
- `weekly_schedules`: 曜日別+祝日の営業時間
- `google_busy`: Googleカレンダーのbusy時間
- `existing_bookings`: 既存のconfirmed予約
- `buffer_before_minutes`: ミーティング前バッファ（app_settingsから取得、初期値30分）
- `buffer_after_minutes`: ミーティング後バッファ（app_settingsから取得、初期値0分）

### 処理

```
[1] 対象日が祝日か判定
    → Google祝日カレンダーのイベントと照合

[2] 営業時間を決定
    → 祝日: weekly_schedules WHERE day_type = 'holiday'
    → 平日: weekly_schedules WHERE day_type = (曜日)
    → is_available = false なら空き0件

[3] 「ブロック時間帯」を生成
    Googleカレンダーのbusy時間と既存予約の両方に対して
    バッファを適用した「使えない時間帯」を算出する

    各busy/予約に対して:
      ブロック開始 = 予定の開始時刻 - buffer_after_minutes  (前の予定の「後バッファ」)
      ブロック終了 = 予定の終了時刻 + buffer_before_minutes  (次の予定の「前バッファ」)

    例: 前バッファ30分、後バッファ0分、既存予定13:00-14:00の場合
      → ブロック時間帯: 13:00〜14:30
      → 14:30から予約可能

[4] 営業時間を30分スロットに分割
    例: 10:00-18:00 → [10:00, 10:30, 11:00, ..., 17:30]
    ※各スロットはメニューの最小時間（30分）単位

[5] 各スロットについて、利用可能なメニューを判定
    例: 10:00のスロット、60分セッションの場合
    → 10:00-11:00がブロック時間帯と重複しない → 60分セッション可
    → 重複する → 60分セッション不可
    ※セッション自体の前後にもバッファを考慮:
      実際に必要な時間帯 = (start - buffer_after) 〜 (end + buffer_before)
      ただし、営業時間の最初のスロットでは前バッファ不要
      営業時間の最後のスロットでは後バッファ不要

[6] ブロック時間帯と重複するスロットを除外
```

### 出力

```json
{
  "date": "2026-03-01",
  "is_holiday": false,
  "slots": [
    {
      "start_time": "2026-03-01T10:00:00+09:00",
      "available_menus": [
        { "id": "...", "name": "カジュアル30分", "duration": 30, "point_cost": 0 },
        { "id": "...", "name": "スタンダード60分", "duration": 60, "point_cost": 60 }
      ]
    },
    {
      "start_time": "2026-03-01T10:30:00+09:00",
      "available_menus": [...]
    }
  ]
}
```

## 6. タイムゾーン方針

| レイヤー | タイムゾーン |
|---|---|
| DB（Supabase） | UTC（`TIMESTAMPTZ`型で保存） |
| API（入出力） | ISO 8601（タイムゾーン付き） |
| フロント表示 | JST（Asia/Tokyo） |
| Google Calendar API | `timeZone: 'Asia/Tokyo'` 指定 |
| Zoom API | `timezone: 'Asia/Tokyo'` 指定 |
| cron（Edge Functions） | JST基準でスケジュール |
