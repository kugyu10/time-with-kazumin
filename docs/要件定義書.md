## 3. API設計

### 3.1 方針

| 処理種別 | 実装場所 | 理由 |
|---|---|---|
| 単純CRUD（RLSで保護可能） | Supabaseクライアント直接 | Route Handler不要でシンプル |
| 外部API連携・複合処理 | Next.js Route Handlers | シークレット管理が容易 |
| cron処理 | Supabase Edge Functions | スケジュール実行に適切 |

### 3.2 Route Handlers

#### 予約系

| メソッド | パス | 説明 | 認証 |
|---|---|---|---|
| GET | `/api/slots` | 指定日の空きスロット取得（同期トリガー含む） | 不要 |
| POST | `/api/bookings` | 予約作成（ポイント消費 + Zoom生成 + メール送信 + カレンダー追加） | 会員 or ゲスト |
| DELETE | `/api/bookings/[id]` | キャンセル（ポイント返還 + Zoom削除 + メール送信 + カレンダー削除） | 本人 or admin |

#### 管理者系

| メソッド | パス | 説明 |
|---|---|---|
| POST | `/api/admin/members` | 会員登録（Supabase Auth招待 + profiles作成 + プラン割当） |
| DELETE | `/api/admin/members/[id]` | 会員退会処理 |
| POST | `/api/admin/points` | ポイント手動調整 |

#### 外部連携系

| メソッド | パス | 説明 |
|---|---|---|
| POST | `/api/integrations/google-calendar/sync` | Googleカレンダー同期（オンデマンド） |

#### cron（Supabase Edge Functions）

| 関数名 | スケジュール | 説明 |
|---|---|---|
| `monthly-point-grant` | 毎月1日 0:00 JST | 月次ポイント自動付与 |
| `send-reminder` | 毎日 9:00 JST | 翌日の予約リマインダー送信 |
| `send-thankyou` | 毎時 0分, 30分 | 終了時刻+30分を過ぎた予約のサンキューメール送信 |

### 3.3 Supabaseクライアント直接アクセス（RLS保護）

以下はRoute Handlerを経由せず、フロントからSupabaseクライアントで直接操作する。

- `meeting_menus` の読み取り（全員）
- `plans` の読み取り（全員）
- `weekly_schedules` の読み取り（全員）/ 書き込み（admin）
- `member_plans` の読み取り（本人 or admin）
- `point_transactions` の読み取り（本人 or admin）
- `bookings` の読み取り（本人 or admin）
- `meeting_menus` のCRUD（admin）
- `plans` のCRUD（admin）
- `app_settings` の読み取り（全員）/ 書き込み（admin）

## 4. データモデル設計

### 4.1 テーブル一覧

全8テーブル + Supabase Auth の `auth.users` を活用。

| テーブル | 概要 |
|---|---|
| `profiles` | ユーザープロフィール（auth.usersの拡張） |
| `plans` | サブスクプラン定義 |
| `member_plans` | 会員のプラン割当 + ポイント残高 |
| `meeting_menus` | メニュー定義（セッション種別） |
| `weekly_schedules` | 曜日別・祝日の営業時間 |
| `bookings` | 予約 |
| `point_transactions` | ポイント履歴 |
| `app_settings` | アプリ全体の設定（バッファ時間、同期タイムスタンプ等） |

### 4.2 各テーブルのスキーマ

#### profiles（ユーザープロフィール）

```sql
CREATE TABLE profiles (
  id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  role TEXT NOT NULL DEFAULT 'guest' CHECK (role IN ('admin', 'member', 'guest')),
  display_name TEXT NOT NULL,
  email TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

- `auth.users` と1対1。`role` でアプリ内権限を管理
- 管理者（かずみん+うおじゅん）は初期データとしてseed

#### plans（サブスクプラン定義）

```sql
CREATE TABLE plans (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,              -- 例: 'ライトプラン'
  monthly_points INT NOT NULL,     -- 毎月付与ポイント数
  is_active BOOLEAN NOT NULL DEFAULT true,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

- MVPでは決済連携なし。プランは「名前」と「月次付与ポイント数」のみ
- `is_active` で論理削除

#### member_plans（会員のプラン割当）

```sql
CREATE TABLE member_plans (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  plan_id UUID NOT NULL REFERENCES plans(id),
  point_balance INT NOT NULL DEFAULT 0,
  started_at DATE NOT NULL DEFAULT CURRENT_DATE,
  ended_at DATE,                   -- NULL = 有効中
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

- 1ユーザー = 1アクティブプラン（`ended_at IS NULL` で絞る）
- `point_balance` を直接保持し、残高取得を1クエリで完結（KISS）
- プラン変更時は旧レコードの `ended_at` を埋めて新レコードを作る

#### meeting_menus（メニュー）

```sql
CREATE TABLE meeting_menus (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,                -- 例: 'カジュアル30分'
  duration_minutes INT NOT NULL,     -- 30, 60, 90, 120
  point_cost INT NOT NULL DEFAULT 0,
  require_membership BOOLEAN NOT NULL DEFAULT false,
  zoom_account_key TEXT NOT NULL DEFAULT 'A' CHECK (zoom_account_key IN ('A', 'B')),
  send_thankyou_email BOOLEAN NOT NULL DEFAULT true,
  is_active BOOLEAN NOT NULL DEFAULT true,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

- `require_membership`: ゲスト可否を制御（メニューごとに設定）
- `zoom_account_key`: 'A'（有料）or 'B'（無料）。環境変数の対応するキーを参照
- `send_thankyou_email`: サンキューメールのON/OFF（メニューごとに設定）

#### weekly_schedules（曜日別・祝日の営業時間）

```sql
CREATE TABLE weekly_schedules (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  day_type TEXT NOT NULL CHECK (day_type IN (
    'monday', 'tuesday', 'wednesday', 'thursday', 'friday', 'saturday', 'sunday', 'holiday'
  )),
  start_time TIME NOT NULL,        -- 例: '10:00'
  end_time TIME NOT NULL,          -- 例: '21:00'
  is_available BOOLEAN NOT NULL DEFAULT true,  -- 休日ならfalse
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  CONSTRAINT valid_schedule_time CHECK (end_time > start_time),
  CONSTRAINT unique_day_type UNIQUE (day_type)
);
```

- 曜日7つ + 祝日1パターン = 最大8レコード
- `holiday` は祝日の営業時間。祝日に該当する日は曜日設定を上書き
- `is_available = false` で「その曜日/祝日は休み」を表現

#### bookings（予約）

```sql
CREATE TABLE bookings (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES profiles(id),
  guest_name TEXT,
  guest_email TEXT,
  meeting_menu_id UUID NOT NULL REFERENCES meeting_menus(id),
  start_time TIMESTAMPTZ NOT NULL,
  end_time TIMESTAMPTZ NOT NULL,
  status TEXT NOT NULL DEFAULT 'confirmed' CHECK (status IN ('confirmed', 'cancelled', 'completed')),
  zoom_meeting_url TEXT,
  zoom_meeting_id TEXT,
  zoom_account_key TEXT,           -- 生成時に使用したZoomアカウント
  google_event_id TEXT,            -- 管理者カレンダーのイベントID（削除用）
  points_consumed INT NOT NULL DEFAULT 0,
  thankyou_sent BOOLEAN NOT NULL DEFAULT false,
  note TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  CONSTRAINT valid_booking_time CHECK (end_time > start_time)
);
```

- 会員予約: `user_id` にプロフィールID
- ゲスト予約: `user_id` は NULL、`guest_name` / `guest_email` に入力値
- `zoom_account_key`: 予約時にどちらのアカウントで生成したか記録（削除時に必要）
- `google_event_id`: 管理者カレンダーのイベントID（キャンセル時の削除用）
- `thankyou_sent`: サンキューメール送信済みフラグ（二重送信防止）

#### point_transactions（ポイント履歴）

```sql
CREATE TABLE point_transactions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  amount INT NOT NULL,              -- 正=付与、負=消費
  type TEXT NOT NULL CHECK (type IN ('monthly_grant', 'booking', 'cancellation', 'manual_adjust')),
  booking_id UUID REFERENCES bookings(id),
  note TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

- 全てのポイント増減を記録する監査ログ
- `type`: 月次付与 / 予約消費 / キャンセル返還 / 管理者手動調整

#### app_settings（アプリ全体の設定）

```sql
CREATE TABLE app_settings (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL,
  description TEXT,
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

- key-value形式の汎用設定テーブル
- 管理画面から変更可能な設定値を格納

**初期データ:**

| key | value | description |
|---|---|---|
| `buffer_before_minutes` | `30` | ミーティング前バッファ（分） |
| `buffer_after_minutes` | `0` | ミーティング後バッファ（分） |
| `last_calendar_sync` | *(空)* | Googleカレンダー最終同期日時 |

### 4.4 ポイント操作のPostgreSQL関数

ポイント操作はトランザクション保証のためにPostgreSQL関数でラップする。

#### consume_points（ポイント消費）

```sql
CREATE OR REPLACE FUNCTION consume_points(
  p_user_id UUID,
  p_amount INT,
  p_booking_id UUID
) RETURNS BOOLEAN AS $$
DECLARE
  current_balance INT;
BEGIN
  SELECT point_balance INTO current_balance
  FROM member_plans
  WHERE user_id = p_user_id AND ended_at IS NULL
  FOR UPDATE;

  IF current_balance IS NULL OR current_balance < p_amount THEN
    RETURN FALSE;
  END IF;

  UPDATE member_plans
  SET point_balance = point_balance - p_amount, updated_at = now()
  WHERE user_id = p_user_id AND ended_at IS NULL;

  INSERT INTO point_transactions (user_id, amount, type, booking_id)
  VALUES (p_user_id, -p_amount, 'booking', p_booking_id);

  RETURN TRUE;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

#### refund_points（ポイント返還）

```sql
CREATE OR REPLACE FUNCTION refund_points(
  p_user_id UUID,
  p_amount INT,
  p_booking_id UUID
) RETURNS VOID AS $$
BEGIN
  UPDATE member_plans
  SET point_balance = point_balance + p_amount, updated_at = now()
  WHERE user_id = p_user_id AND ended_at IS NULL;

  INSERT INTO point_transactions (user_id, amount, type, booking_id)
  VALUES (p_user_id, p_amount, 'cancellation', p_booking_id);
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

## 5. 認証・権限設計

### 5.1 認証フロー

```
[ゲスト]  --- 予約フォームにメール+名前入力 ---> 予約作成（auth不要）
[会員]   --- Google or メール/パスワード -----> Supabase Auth → profiles.role = 'member'
[管理者] --- 同上 ----------------------------> Supabase Auth → profiles.role = 'admin'
```

### 5.2 会員招待フロー

1. 管理者が `/admin/members` で「新規会員追加」
2. `supabase.auth.admin.inviteUserByEmail()` で招待メール送信
3. `profiles` テーブルに `role = 'member'` でレコード作成
4. `member_plans` にプラン割当 + 初期ポイント付与
5. ユーザーが招待メールのリンクからパスワード設定 → ログイン可能に

### 5.4 ゲスト予約のアーキテクチャ

ゲスト予約は認証なしで行うため、RLSでは保護できない。

- 予約作成は全て `/api/bookings` Route Handler 経由
- Route Handler内で `service_role` キーを使ってSupabaseに書き込み
- フロントからSupabaseに直接書き込ませない
- Rate Limiting（Vercel Edge Middleware or Upstash）でスパム対策
